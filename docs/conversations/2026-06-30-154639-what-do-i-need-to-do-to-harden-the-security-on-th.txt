‚Źļ Good news first: this server is already hard to abuse. Every route runs

  /edits will buffer whatever an allowlisted-but-misbehaving client sends. One

  2. The AI prompt-injection "defense" is theater. Lines 127-129 add a trailing
  system message telling the model to ignore malicious instructions ‚ÄĒ that
  doesn't reliably stop injection. But it doesn't need to: the brace-stripping
  on line 159 + selector-wrapping on 161 is the real containment. The output is
  forced to be CSS declarations inside a caller-fixed selector. Keep line 159, 
  drop the false-comfort prompt or keep it, doesn't matter ‚ÄĒ the structural
  defense is what holds. Worth a comment that the brace strip is the actual
  boundary, not the prompt.

  3. Rate limit only covers /ai/edit. That's the expensive endpoint, so fine.
  But /state and /edits writes are unbounded per user ‚Üí a buggy/malicious
  allowlisted client can fill KV/D1. If you care, move RL.limit up into
  authenticate (global per-user budget). Probably YAGNI for an allowlisted user
  base ‚ÄĒ skip until you have more than friends on the list.

  Cloudflare side ‚ÄĒ where scraper/abuse hardening actually lives

  The Worker can't protect itself from volume; the edge does. In the dashboard:

  1. Confirm Access is enforced at the zone, not just in-app. Right now the only
  thing stopping an unauthenticated flood is your jwtVerify running per-request

  4. robots.txt is advisory only ‚ÄĒ the toggle above is what enforces it.

  Cloudflare Dashboard Hardening Checklist

  Do these once in the dashboard. Items 1‚Äď2 are the load-bearing ones; the rest
  are cheap extra layers.

  1. Confirm Cloudflare Access fronts the Worker at the edge

  Right now unauthenticated requests still cost you a Worker invocation (your
  jwtVerify runs per request). Push rejection to the edge.

  1. Zero Trust dashboard ‚Üí Access ‚Üí Applications.
  2. Find the app whose AUD matches
  d3806e652e3f20a509d8eca588b660accb22080fd302e84116dfee377db4c3d2 (your
  ACCESS_AUD). If none exists, Add an application ‚Üí Self-hosted.
  3. Application domain: set to the Worker's hostname + path (e.g.
  frimmy.johnmemmott.workers.dev, path *).
  4. Policies: one policy, action Allow, rule Emails ‚Üí your allowlist (the same
  addresses in the ALLOWED_EMAILS secret). Everything else is implicitly
  blocked.
  5. Save. Now unauthenticated traffic is challenged/blocked before it reaches
  the Worker.

  2. Block AI scrapers and bots

  1. Main Cloudflare dashboard ‚Üí select your zone ‚Üí Security ‚Üí Bots (or Security
  ‚Üí Settings).
  2. Toggle Bot Fight Mode ON (free tier). This blocks known bad bots by
  behavior + UA.
  3. Find Block AI bots / "Block AI Scrapers and Crawlers" and set it to Block
  (covers GPTBot, ClaudeBot, CCBot, etc. ‚ÄĒ enforced, unlike robots.txt).
  4. Optional: AI Audit tab to see which AI crawlers have hit the zone.

  3. Rate limiting at the edge

  Backstop your per-user RL binding with a per-IP edge limit.
  1. Security ‚Üí WAF ‚Üí Rate limiting rules ‚Üí Create rule.
  2. If incoming requests match: Hostname equals your Worker host (optionally
  path /ai/*).
  3. Rate: e.g. 50 requests per 1 minute, Per: IP.
  4. Action: Block (or Managed Challenge), duration 1 min. Deploy.

  4. WAF managed rules

  1. Security ‚Üí WAF ‚Üí Managed rules.
  2. Enable Cloudflare Managed Ruleset (and OWASP Core if available on your
  plan). Default sensitivity is fine.

  5. Lock CORS realism (optional)

  Your Worker sends access-control-allow-origin: *. That's safe because you auth
  via header, not cookies ‚ÄĒ but if you ever want fewer probes, you can leave
  it. No action needed unless you switch to cookie auth.

  6. robots.txt (advisory only)

  Not enforcement ‚ÄĒ item 2 is. Skip unless you want polite crawlers to
  self-exclude; the Worker has no static route for it anyway.
