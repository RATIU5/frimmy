‚Źļ Already works. No schema change needed.

  The flow handles multiple requests to the same element by merge-via-context,
  not multi-edit JSON:

  - edits is keyed by selector (App.tsx:199). A second request to the same
  element overwrites the same key.
  - Before each request, the prior CSS for that selector is fed back to the AI
  with "modify/extend it, keep existing declarations" (App.tsx:177-182).
  - The model returns the full merged declaration block, so {declarations:
  string} stays single ‚ÄĒ it just keeps growing.
  - All selectors' rules are concatenated on save (App.tsx:208).

  So N requests to one element ‚Üí one accumulating rule. N requests across
  different elements ‚Üí N keyed rules. Both covered.

  The one real weakness is reliability, not architecture: the 8B model can drop
  prior declarations despite the instruction. If you see edits getting
  clobbered, the fix is a deterministic merge (parse prior + new declarations,
  new wins per-property) instead of trusting the model ‚ÄĒ but that's only worth
  doing if you actually observe it.

  skipped: multi-edit JSON schema + deterministic declaration merge. add when
  you observe the model dropping prior declarations in practice.
