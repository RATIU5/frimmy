‚Źļ Found it. There's a race between the two restore effects that wipes the chat.

  Both effects run on mount and both call setMessages:
  - The ?frimmy effect sets messages from the shared blob.
  - The load-state effect does if (Array.isArray(s?.messages))
  setMessages(s.messages) ‚ÄĒ and [] is an array. So if this URL has a saved state
  whose messages is empty (or it resolves second), it overwrites the restored
  conversation with []. Same clobber risk for edits (non-functional
  setEdits(s.edits) vs the frimmy effect's functional merge).

  Whichever resolves last wins, and an empty array silently beats a populated
  one ‚Üí chat box ends up empty.

  Fix: sequence them ‚ÄĒ load per-URL state first, then overlay the shared blob ‚ÄĒ
  and never let an empty restore clobber existing messages.
