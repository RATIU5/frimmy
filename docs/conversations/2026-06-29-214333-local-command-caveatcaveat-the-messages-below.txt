⏺ Heads up before the steps: your two docs disagree. SPEC_SERVER.md describes
  OAuth + self-issued JWT; README.md describes Cloudflare Access. The README is
  the lazier, correct call (no OAuth routes, no JWT signing, no refresh logic —
  Cloudflare does it). I'll walk you through Access. If you actually want the
  OAuth path, say so.

  Cloudflare dashboard — setting up Access

  Prereq: your Worker must be deployed on a route/custom domain (Access protects
  hostnames/paths, and it can't sit in front of a plain *.workers.dev route —
  you need the Worker on a domain in your CF account). So wrangler deploy with a
  route first.

  1. Turn on Zero Trust

  - Dashboard → Zero Trust (left sidebar). First time, it'll make you pick a
  team name → that's your <team> in https://<team>.cloudflareaccess.com. Free
  plan covers up to 50 users.

  2. Add a login method (identity provider)

  - Zero Trust → Settings → Authentication → Login methods.
  - Add Google (or GitHub). The built-in One-time PIN works with zero config if
  you just want email-code login to start — pick that to skip OAuth app setup
  entirely (lazy default).

  3. Create the Access application

  - Zero Trust → Access → Applications → Add an application → Self-hosted.
  - Application domain: the Worker's hostname + path, e.g. api.yourdomain.com
  and path / (or scope to a subpath).
  - Identity providers: select the ones from step 2.

  4. Add a policy

  - In the same app, Add policy:
    - Name: allow-me
    - Action: Allow
    - Include → Emails → your email (jasonmemm@gmail.com).
  - Save. This is what restricts access to you.

  5. Grab the two values for wrangler.jsonc

  - ACCESS_TEAM_DOMAIN = https://<team>.cloudflareaccess.com (from step 1).
  - ACCESS_AUD = the application's Audience (AUD) Tag → on the app's Overview
  tab, copy the AUD tag.

  Put both in wrangler.jsonc → vars. The Worker verifies the
  Cf-Access-Jwt-Assertion header against
  <team>.cloudflareaccess.com/cdn-cgi/access/certs with jose.

  ---
  The one real gotcha for an extension (not a dashboard step): Access normally
  authenticates via an interactive browser login that drops a cookie on the
  protected hostname. A browser-extension fetch won't carry that automatically.
  Two options:
  - Service token (Access → Service Auth → Service Tokens): extension sends
  CF-Access-Client-Id + CF-Access-Client-Secret headers. Simplest for a
  solo/personal extension, but the secret lives in the extension — fine for just
  you, not for a public release.
  - launchWebAuthFlow against the Access login URL to get a user JWT, then send
  it as Cf-Access-Jwt-Assertion. Correct for multi-user, more work.

  For your solo stage, a service token is the lazy answer. Want me to wire up
  the Worker-side JWT verification (jose against the certs endpoint) once the
  dashboard side is done?
