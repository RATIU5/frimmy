‚ĚĮ better to not use tailwind when using shadow dom?

‚Źļ Honest answer: Tailwind works in shadow DOM but it's the awkward fit, and for
  one panel it's not worth it. The friction points:

  - Resets leak/miss. Tailwind's Preflight is written for :root/html; in a
  shadow root you have to re-scope it or styles come out half-applied.
  - rem units. Tailwind sizes in rem, which resolve against the page's root
  font-size, not your shadow tree ‚ÄĒ so a site with html { font-size: 8px }
  wrecks your panel. You'd set a fixed font-size on the host to defend against
  it.
  - v4 @theme/inject config into the shadow root adds setup you'd otherwise
  skip.

  Shadow DOM already gives you the thing Tailwind's utility approach is partly
  substituting for: style isolation. Plain scoped CSS in the shadow root can't
  leak out and the page can't leak in. For a chat panel + picker overlay that's
  a handful of classes ‚ÄĒ plain CSS is fewer moving parts and zero of the above
  gotchas.

  So: plain CSS, co-located with the content UI. No new dep (drop the tailwind
  install). If the component count ever explodes, add Tailwind then with the
  host-scoped reset ‚ÄĒ but YAGNI for one panel.
