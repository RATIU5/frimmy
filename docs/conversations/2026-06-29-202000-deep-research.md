# Building a Cloudflare-Native REST API for a Browser-Extension Backend (2025–2026)

## TL;DR

- **The stack is sound and production-ready.** Use Hono on a single Worker, D1 for metadata/ownership, KV for diff blobs (with R2 as the fallback above ~1 MB), Workers AI (JSON Mode) gated by the GA `ratelimit` binding plus an AI Gateway spend cap, and `jose` for stateless JWTs. Every component here is GA as of mid-2026.
- **The two highest-risk areas are cost runaway on Workers AI and the dual-write D1↔KV consistency gap.** Cap AI spend with AI Gateway dollar limits (open beta for all gateways/plans since June 5, 2026), cache AI responses in KV, rate-limit per authenticated user, and write D1 metadata last so a KV blob is never orphaned by a failed metadata commit.
- **Bot Fight Mode will silently break your extension API.** It runs outside the Ruleset Engine and cannot be skipped — use Super Bot Fight Mode (Pro+) with a Skip rule on your `/api` path, or keep the API on a hostname where BFM is off, and use the Workers `ratelimit` binding rather than WAF rate limiting for per-user enforcement.

## Key Findings

1. **Hono is the clear router choice.** It is ~12–14 KB gzipped, the fastest JS-world router, has first-class TypeScript generics for typing `env` bindings, and is used internally by Cloudflare for D1, KV and Queues. itty-router (~500 bytes) only wins if bundle size is critical and you need no middleware.
2. **D1 is GA but capped at 10 GB per database (hard, architectural limit).** Per Cloudflare's D1 release notes and Limits docs: Free plan = 500 MB/db and up to 10 dbs; Paid = 10 GB/db combined with 50,000 dbs/account; and "the 10 GB limit of a D1 database cannot be further increased." No interactive transactions (`BEGIN/COMMIT` throws) — use `db.batch()` for atomic multi-statement work. Single-writer: per the docs, "If your average query takes 1 ms, you can run approximately 1,000 queries per second."
3. **KV is eventually consistent and rate-limits writes.** Per Cloudflare's "How KV works" docs, changes "may take up to 60 seconds or more to be visible in other global network locations," and negative lookups are cached too. KV "has a maximum of 1 write to the same key per second," and the maximum value size is 25 MiB. Reads: ~5–15 ms hot, 50–100 ms+ cold. Never use KV for counters or rate limiting.
4. **Workers AI: $0.011 per 1,000 Neurons, 10,000 Neurons/day free.** Per the Workers AI pricing docs: "priced at $0.011 per 1,000 Neurons. Our free allocation allows anyone to use a total of 10,000 Neurons per day at no charge... All limits reset daily at 00:00 UTC." Most LLMs are capped at 300 req/min. JSON Mode (OpenAI-compatible `response_format`) is GA. Edge deployment gives no real latency benefit for inference — model choice is the dominant cost and latency lever.
5. **Stateless JWT has no revocation.** Use short access-token TTLs (5–15 min), keep refresh logic server-side, and accept that "logout" = client-side token deletion + expiry, unless you add a KV/D1 blocklist.
6. **Workers Rate Limiting binding is GA.** Per Cloudflare's changelog dated 2025-09-19: "Rate Limiting within Cloudflare Workers is now Generally Available (GA). The ratelimit binding is now stable and recommended for all production workloads. Existing deployments using the unsafe binding will continue to function." Use it (not the legacy `unsafe` binding) keyed on user ID for per-user AI quotas. Note: counters are per-Cloudflare-location, not global.
7. **chrome-extension:// origins work in CORS** — set `Access-Control-Allow-Origin` to the specific extension origin and handle the OPTIONS preflight (200/204, no redirect). MV3 service-worker `fetch` from the background context honors `host_permissions`.

## Details

### Workers (routing, lifecycle, errors, CORS)

**Router.** Hono is the 2025/2026 default for Workers. Bundle: hono/tiny ~12 KB, default ~12–18 KB gzipped, zero dependencies, Web-Standards only. It beats itty-router and worktop on throughput (Hono ~402K ops/s vs itty ~212K in the maintainers' benchmark) and ships typed middleware, `zValidator` (Zod) request validation, and an RPC client for end-to-end types. itty-router is ~500 bytes and uses ~0.7 ms CPU vs Hono's ~2.6 ms in one community test — relevant only at extreme cost-optimization. Recommendation: **Hono**.

Type your bindings via a generic:

```ts
type Bindings = {
  DB: D1Database;
  DIFFS: KVNamespace;
  AI: Ai;
  RL: RateLimit; // ratelimit binding
  JWT_SECRET: string;
};
const app = new Hono<{ Bindings: Bindings; Variables: { userId: string } }>();
```

Run `wrangler types` to generate runtime types for all bindings.

**Lifecycle & limits (current).**

- **CPU time:** 50 ms default on many plans; Paid Standard can raise the per-invocation CPU limit up to **5 minutes (300,000 ms)**. Per the Workers Limits docs, "The average Worker uses approximately 2.2 ms per request. Heavier workloads that handle authentication, server-side rendering, or parse large payloads typically use 10-20 ms." Exceeding the CPU limit → Error 1102.
- **Wall-clock:** no hard duration limit for HTTP Workers while the client is connected; `ctx.waitUntil()` can extend work up to 30 s after response.
- **Subrequests:** up to **1,000** external operations per invocation; **6 simultaneous connections** waiting for response headers (a 7th queues).
- **Startup:** global scope must parse/execute within 1 s (Error 10021 otherwise).
- **Bundle:** 1 MB compressed (Free), 10 MB (Paid). Store assets in KV/R2/D1, not the bundle.

**Structured errors.** Centralize with Hono's `app.onError()`; return a typed JSON envelope (`{ error: { code, message, requestId } }`) and map domain errors to HTTP codes. Never leak D1/AI internals to clients.

**CORS for `chrome-extension://`.** This is the most misunderstood point: `Access-Control-Allow-Origin` **does** accept a `chrome-extension://<id>` value. Echo the specific extension origin (do not use `*` if you send credentials), and answer the preflight `OPTIONS` with a 2xx (200/204) — a redirect on preflight fails. Use Hono's `cors()` middleware with an allowlist:

```ts
app.use(
  "/api/*",
  cors({
    origin: ["chrome-extension://abcdef…", "chrome-extension://<firefox-id>"],
    allowMethods: ["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allowHeaders: ["Authorization", "Content-Type", "Idempotency-Key"],
    maxAge: 86400,
  }),
);
```

MV3 background service workers send the extension origin in the `Origin` header and require the Workers domain in `host_permissions`. Since Chrome 85, content-script cross-origin fetches are subject to CORS too — make API calls from the background service worker, not content scripts.

### D1

- **GA since Developer Week 2024.** Built on SQLite, each DB backed by a single Durable Object.
- **Limits:** 10 GB/db (hard, cannot be raised), 50,000 dbs/account (Paid), 6 simultaneous D1 connections per Worker invocation. Free tier = 500 MB/db and up to 10 dbs, with daily caps (5M rows read / 100K written) resetting at 00:00 UTC.
- **No interactive transactions.** `BEGIN TRANSACTION` / Drizzle `db.transaction()` throw. Use `db.batch([stmt, stmt])` — statements run sequentially, non-concurrently, auto-commit, and roll back as a unit if one fails.
- **Prepared statements:** `db.prepare().bind()` with ordered `?` / `?NNN` params (named params not yet supported). `.run()`, `.first()`, `.all()`, `.raw()`. Generic typing: `prepare<OrderRow>()`.
- **Performance:** indexed point reads <1 ms SQL duration; writes several ms. Single-writer ⇒ throughput ≈ 1/avg-query-time (1 ms ⇒ ~1,000 q/s). Overloaded DBs queue, then return "overloaded" errors — implement backpressure.
- **Migrations:** `wrangler d1 migrations create` / `apply`; since wrangler 3.33 these default to **local** — pass `--remote` for prod. Keep a migrations dir per environment.
- **Don't store large blobs in D1** (row-size limit ~2 MB; community reports of "row size exceeded" failures). Put diff blobs in KV/R2 and only the metadata + key in D1.
- **When D1 is wrong:** need >10 GB single logical DB without sharding, heavy concurrent writes, true Postgres features → use Hyperdrive + Neon/Postgres. Need strong per-entity consistency / high write rate on one key → Durable Objects (SQLite backend).

### Workers KV (diff blobs)

- **Eventual consistency:** writes are immediately visible in the writing location but, per the docs, "may take up to 60 seconds or more to be visible in other global network locations as their cached versions of the data time out." Negative (key-not-found) lookups are cached too.
- **Write ceiling:** maximum 1 write to the same key per second — hot-key writes get rejected. KV is **not** for counters/rate-limiting.
- **Value size:** 25 MiB; key ≤ 512 B; metadata ≤ 1,024 B. `cacheTtl` minimum 60 s.
- **Latency:** ~5–15 ms hot, 50–100 ms+ on cold first read in a colo; the 2025 re-architecture cut KV latency materially by connecting Workers directly to KV.
- **Key design for diffs:** key by edit ID (`diff:<ulid>`); store small display metadata in KV metadata to avoid extra reads; consider `edit:<id>:v<n>` for versions. Diffs are write-once/read-many — an ideal KV fit.
- **Pricing (Paid):** reads 10M/mo included then $0.50/M; writes/deletes/lists 1M/mo then $5/M; storage 1 GB then $0.50/GB-mo. Every op (incl. 404s) is billable.

### KV vs R2 vs D1 for blobs

- **KV:** best for small (<1 MB), write-once, read-heavy, globally cached diffs. Cheap reads, pricey writes.
- **R2:** better for large blobs (>1 MB, up to GB), no egress fees, cheaper per-GB storage, strong read-after-write within a bucket. The community "store-in-R2-above-1MB" pattern is the standard hybrid.
- **D1:** only metadata and the blob's KV/R2 key — never the blob itself.

### JWT / Auth on Workers

- **Use `jose`** (panva): zero deps, tree-shakeable ESM, explicitly supports Cloudflare Workers via WebCrypto, ~88M weekly downloads. `hono/jwt` is fine for simple HS256 in-framework verification but `jose` is the more complete/correct choice (JWKS, RS256/ES256, `createRemoteJWKSet` with caching). No Node `crypto` needed — both use WebCrypto.
- **Key management:** store signing secret/private key via `wrangler secret put` (and `.dev.vars` for local). For asymmetric (RS256/ES256), keep the private key secret-side and expose a JWKS endpoint so verification can rotate.
- **Rotation:** publish multiple keys in JWKS with `kid`; sign with the new key while still accepting the old until all old tokens expire.
- **Stateless pitfalls:** no server-side revocation. Mitigations: short access TTL (5–15 min) + refresh token held by the extension; optional KV/D1 token-blocklist (`jti`) if you need immediate revocation (reintroduces a read per request, breaking pure statelessness).

### OAuth for the extension (MV3)

- **Use `chrome.identity.launchWebAuthFlow`**, not `getAuthToken` — `getAuthToken` is Chrome-only and fails in Brave/Edge/other Chromium browsers. `launchWebAuthFlow` works cross-browser (and Firefox `browser.identity`).
- **Flow:** redirect URI = `https://<extension-id>.chromiumapp.org/` (via `chrome.identity.getRedirectURL()`), use Authorization Code + PKCE with `response_type=code`, exchange the code on your Worker (keep the client secret server-side; never in the extension), and return your own app JWT. The popup closes on login — run the flow from the background service worker and persist results to `chrome.storage`.
- **Token storage:** `chrome.storage.session` (in-memory, cleared on browser close, not on disk) is safer for access tokens; `chrome.storage.local` persists across restarts (use for the refresh token, accepting the higher exposure). Neither is readable by web pages, but other extensions/malware with debugger access can reach them — keep TTLs short.
- **Logout/revocation:** delete tokens from `chrome.storage` client-side; tokens expire server-side. True revocation needs a blocklist.

### Security: WAF / Bot Fight Mode / AI-crawler blocking

- **Bot Fight Mode (Free) cannot be skipped** — it runs outside the Ruleset Engine, so WAF Skip/Allow/Page Rules don't affect it, and it has repeatedly broken legitimate API/non-browser traffic (well-documented community reports). Options: (a) upgrade to **Super Bot Fight Mode** (Pro+) which runs on the Ruleset Engine and supports Skip rules — add a Skip on `starts_with(http.request.uri.path, "/api")` or your extension's JA3/identifier; or (b) host the API on a hostname/zone where BFM is off.
- **AI-crawler blocking ("Block AI bots")** takes precedence over other SBFM rules; ensure your `/api` path isn't caught — scope it out or rely on the fact that authenticated extension traffic carries a Bearer token (add a custom rule allowing requests with a valid `Authorization` header pattern, but enforce real auth in the Worker).
- **Rate limiting choice:** use the **Workers `ratelimit` binding** for per-user/in-code limits (it's fast, memcached-backed, no added latency, keyed on whatever you choose). Use **WAF rate-limiting rules** for coarse edge/IP-level abuse protection. Note the WAF gotcha: same-zone Worker subrequests can be double-counted unless you exclude `cf.worker.upstream_zone`.

### Wrangler / Bun / CI-CD

- `wrangler dev` emulates D1, KV, and the AI binding locally (Miniflare). Secrets for local dev go in `.dev.vars`.
- WXT (Vite/Bun) runs its own dev server for the extension; run `wrangler dev` separately for the API — they're independent processes. Point the extension's dev build at `http://localhost:8787`.
- **Environments:** define `[env.staging]` / `[env.production]` blocks in `wrangler.toml` (now `wrangler.jsonc` supported) with separate D1/KV IDs; deploy with `wrangler deploy --env production`.
- **CI/CD:** GitHub Actions + `cloudflare/wrangler-action` (or `npx wrangler deploy`), with `CLOUDFLARE_API_TOKEN` as a repo secret; run `wrangler d1 migrations apply --remote --env production` as a deploy step.
- **Rate-limit binding config** (post-GA): the GA `ratelimit` binding is stable; existing `unsafe.bindings` still works for transition. The Wrangler binding schema for `ratelimit` has been criticized as inconsistent (docs issue #25590, workers-sdk #10447) — verify against current docs.

## Integration Guidelines

### Dual-write D1 + KV (the core consistency problem)

No distributed transactions exist across D1 and KV. Recommended pattern:

1. **Generate a ULID edit ID client-or-server-side** (sortable, URL-safe, good index locality).
2. **Write the blob to KV first** (`DIFFS.put('diff:<ulid>', blob)`).
3. **Then write metadata to D1 last** as the "commit point." If the D1 write fails, you have an orphan KV blob (harmless, cleaned by a TTL or sweep) but never a metadata row pointing at a missing blob — the dangerous direction.
4. **Idempotency:** require an `Idempotency-Key` header on create; store it in D1 with a unique constraint so retries don't double-create. Because KV `put` is idempotent by key, replaying the same ULID is safe.
5. **Compensation:** a scheduled (Cron) sweep deletes KV blobs with no matching D1 row after N hours.
   Rationale: KV's up-to-60 s propagation means a just-written blob may not be globally visible immediately; design reads to tolerate a brief miss (retry, or read from the writing region). For strong consistency on ownership, a Durable Object per resource is the alternative.

### Workers AI cost control

- **Per-user quota:** use the `ratelimit` binding keyed on `userId` (e.g., 20 AI calls / 60 s). For monthly quotas, a D1 counter row (incremented in the same batch as the edit insert) is more accurate than KV (which can't count reliably).
- **Cache AI responses in KV:** key on a hash of (prompt + page-context + model); serve cache hits for free, skipping inference entirely. This is the single biggest cost lever for repeated/similar diffs.
- **Hard dollar cap:** route Workers AI/3rd-party models through **AI Gateway spend limits**. Per Cloudflare's blog "Your AI bill is out of control. Cloudflare can fix it now." (June 5, 2026): "Spend limits are available in open beta today for all AI Gateway users across all plans." The changelog ("Control AI costs with spend limits," 2026-06-05) adds that spend limits block requests with a 429 until the window resets, can be scoped by "model, provider, or custom metadata dimensions like user ID, team, or application," with up to 20 rules per gateway. Note Unified Billing carries a 5% fee and (in beta) does not cover Workers AI's own neuron billing, which is separate — so also set account billing alerts.
- **Pick the smallest model:** `llama-3.1-8b-instruct-fp8-fast` (4,119/34,868 neurons per M in/out) is ~6× cheaper on input than the standard 8B and far cheaper than the 70B (26,668/204,805). Reserve 70B for genuinely hard diffs.

### Cost model (illustrative: 100K users, 1M AI calls/month)

Assume each diff call ≈ 1,500 input + 500 output tokens on `llama-3.1-8b-instruct-fp8-fast`:

- Neurons/call ≈ (1,500/1e6 × 4,119) + (500/1e6 × 34,868) ≈ 6.18 + 17.43 ≈ **23.6 neurons/call**.
- 1M calls ≈ **23.6M neurons** ≈ minus ~0.3M free/mo ≈ ~23.3M billable × $0.011/1,000 ≈ **~$256/month** for AI (before caching). On the 70B it would be ~$1,200+/mo — model choice ≈ 5× swing.
- KV: 1M writes (diffs) ≈ $5/M ⇒ ~$5 + reads (say 10M) ≈ ~$0 within included + storage. D1: dominated by row reads/writes, typically a few dollars at this scale within/near the $5 Paid plan. Workers requests: 1M+ calls trivially within Paid.
- **Total ballpark: ~$270–300/month** on the small model, dominated by AI; the 70B pushes it past $1,200. Caching a 30% hit rate cuts AI ~30%.

### Extension ↔ Worker CORS/auth checklist (integration)

- Worker echoes the specific `chrome-extension://` origin and handles OPTIONS with 204.
- Extension manifest lists the Workers domain in `host_permissions`; fetches run in the background service worker.
- Bearer JWT in `Authorization`; refresh handled by a dedicated `/auth/refresh` Worker route.
- SBFM Skip rule (or BFM off) on the API hostname so extension traffic isn't challenged.

## Known Issues & Gotchas

- **D1 `BEGIN TRANSACTION` throws**; Drizzle `db.transaction()` errors on D1. Use `batch()`. REST API doesn't support parameterized batch (workerd issue #3749) — use the Worker binding, not REST, for batched writes.
- **D1 10 GB is a hard ceiling** — plan sharding (per-tenant/per-entity) early if you'll exceed it.
- **KV write-after-read across colos returns stale data up to 60 s**; KV hot-key writes (>1/s) rejected. The #1 "KV is broken" support theme is misuse for counters.
- **Rate-limit binding counters are per-Cloudflare-location**, not global — a user routed through multiple POPs can exceed a naive global limit. The binding isn't dashboard-visible (observe via Logs/Analytics Engine). Wrangler binding schema for `ratelimit` is inconsistent/confusing (docs issue #25590, workers-sdk #10447).
- **Bot Fight Mode** silently challenges non-browser/API traffic and can't be skipped — a recurring production outage cause.
- **MV3 service worker** is ephemeral: register event listeners at top level (not in async callbacks), don't keep state in module globals (use `chrome.storage`), no DOM/`localStorage`/`XMLHttpRequest` — use `fetch`. `getAuthToken` is Chrome-only.
- **Workers AI neuron accounting** has had community reports of dashboard counts not matching docs — monitor actual spend, don't trust estimates alone.
- **AI Gateway spend limits are open beta** (June 2026) and don't cover Workers AI's own billing — re-verify before relying on them as your only cap.

## Anti-Patterns (what NOT to do)

- **Global mutable state in a Worker** for per-user/session data — isolates are reused and shared; never cache user data in module scope.
- **Assuming Node APIs** (`fs`, Node `crypto`, `Buffer` semantics) — use WebCrypto, Web Streams, `fetch`.
- **Long-lived connections / holding the event loop** — Workers are request-scoped; use `ctx.waitUntil` for fire-and-forget.
- **Top-level `await` doing heavy init** — counts against the 1 s startup limit; lazy-init inside handlers.
- **KV for counters/rate-limits/transactions** — use the `ratelimit` binding or Durable Objects.
- **Storing diff blobs in D1** — row-size limits will fail on large diffs; use KV/R2 + key in D1.
- **`getAuthToken` for OAuth** — breaks in non-Chrome browsers; use `launchWebAuthFlow`.
- **Client secret in the extension** — do the code exchange server-side.
- **`Access-Control-Allow-Origin: *` with credentials** — invalid; echo the specific extension origin.
- **Writing D1 metadata before the KV blob** — risks metadata pointing at a missing blob.

## Recommendations (staged)

**Stage 1 — MVP (build now):**

1. Single Worker + Hono; type bindings; `app.onError` envelope; Hono `cors()` allowlisting your extension origins.
2. D1 schema via `wrangler d1 migrations`; ownership/edit-metadata tables; ULID PKs; `Idempotency-Key` unique constraint.
3. KV for diff blobs keyed `diff:<ulid>`; write KV-first, D1-last.
4. `launchWebAuthFlow` + PKCE → Worker code exchange → `jose` HS256 (or ES256 + JWKS) app JWT, 15-min TTL + refresh route.
5. `ratelimit` binding per-user on AI and write endpoints.

**Stage 2 — Hardening (before scale):** 6. Workers AI via `llama-3.1-8b-instruct-fp8-fast`, JSON Mode + Zod schema; cache responses in KV; Llama Guard on inputs if accepting arbitrary page content. 7. AI Gateway dollar spend limit + account billing alerts. 8. Super Bot Fight Mode with Skip rule on `/api`; verify "Block AI bots" doesn't hit the API. 9. Observability: Workers Logs + Logpush to R2; Analytics Engine datapoint on `ratelimit` rejections and AI calls. 10. Move blobs >1 MB to R2.

**Stage 3 — Scale (>10 GB D1 or heavy concurrency):** 11. Shard D1 per-tenant/per-entity, or move strong-consistency ownership to Durable Objects. 12. Consider RS256/ES256 + rotating JWKS; add a `jti` blocklist if immediate revocation is required.

**Thresholds that change the plan:** D1 approaching ~7–8 GB → shard. AI bill > caching-adjusted budget → downgrade model / raise cache TTL / tighten per-user quota. p95 inference > UX budget → smaller model or streaming. Repeated 429s from Workers AI (300 req/min) → queue + backoff or request a limit increase.

## Caveats

- Cloudflare's AI, D1, KV and Rate-Limiting surfaces evolve fast; re-verify exact limits/pricing against `developers.cloudflare.com` before launch. Neuron prices cited are from the docs page dated Apr 21, 2026.
- AI Gateway spend limits and identity-driven budgets were in beta as of June 2026 — treat as not-yet-stable.
- The Workers `ratelimit` binding's per-location counting is a real semantic limitation for global quotas; for exact global counts use a Durable Object.
- Cost model figures are illustrative; actual neurons depend on real token counts and model mix.
